## 3. Findings detalhados

### F1 🚨 CRITICAL — Onboarding bloqueia agent loop sem `USER.md`

**Sintoma:**
- Smoke bench passou de 18/20 (baseline) a **2/20 (10 %)** sem nenhuma mudança de código.
- Agente respondeu "What's your role and work context?" em vez de executar a task em todo cenário.

**Causa raiz:**
- `crates/theo-agent-runtime/src/onboarding.rs::compose_bootstrap_system_prompt` injeta `BOOTSTRAP_PROMPT` no system message quando `<repo>/.theo/memory/USER.md` está ausente ou tem < 50 bytes.
- O smoke runner (`apps/theo-benchmark/runner/smoke.py`) cria tmpdir limpo a cada cenário ⇒ sempre cai no caminho de onboarding.
- `THEO_SKIP_ONBOARDING=1` existe como bypass mas o smoke runner NÃO o seta.

**Fix aplicado (workaround):**
- Setar `THEO_SKIP_ONBOARDING=1` antes do smoke ⇒ devolve 19/20 (95 %).
- `USER.md` populado no projeto também desabilita o onboarding.

**Recomendação para produção:**
1. Smoke runner deve setar `THEO_SKIP_ONBOARDING=1` por default (`apps/theo-benchmark/runner/smoke.py` ~ linha 100, antes de `subprocess.run`).
2. CLAUDE.md "Honest System State" deve documentar este pré-requisito do bench reproduzir o baseline.
3. Idealmente, o `--headless` mode deveria implicar `skip_onboarding=true` (CI/bench nunca quer onboarding).

### F2 🚨 HIGH — `AgentLoop::build_registry` ignorava o registry passado em `new()` (FIXED)

**Sintoma:**
- LSP / DAP / Browser tools sempre retornavam "No X servers discovered on PATH" mesmo com sidecars instalados.
- CLAUDE.md "Honest System State" documenta isso como "4 sidecar tool families wired but unexercised".

**Causa raiz:**
- `cmd_headless` (apps/theo-cli/src/main.rs:968) chamava `create_default_registry_with_project(project_dir)` corretamente — esta função faz PATH discovery e popula a `LspSessionManager` com servidores reais.
- O registry era passado para `AgentLoop::new(config, registry)`.
- **MAS** `AgentLoop::build_engine` (crates/theo-agent-runtime/src/agent_loop/mod.rs:399) chamava `self.build_registry(project_dir, ...)` que **DESCARTAVA o registry** e construía um novo via `create_default_registry()` — o que tem `LspSessionManager::from_catalogue(HashMap::new())` (catálogo VAZIO).

**Evidência empírica (instrumentada):**
```
[lsp-discover] discovered 1 servers          ← from_path() encontra rust-analyzer
[lsp_status] Tool::execute called; extensions=0 (manager_ptr=0x...)
                                          ↑ outra instância — vazia
```

**Fix aplicado em `crates/theo-agent-runtime/src/agent_loop/mod.rs`:**
```rust
fn build_registry(...) -> ToolRegistry {
    // BUGFIX 2026-04-27: was create_default_registry() (empty catalogues)
    let mut registry = create_default_registry_with_project(project_dir);
    load_plugin_tools(...);
    registry
}
```

**Validação pós-fix:**
- `lsp_status` agora retorna `"1 routable extension: rs"` com rust-analyzer 1.95.0 detectado.
- `lsp_definition` invocado 3 vezes com `success_count=3, failure_count=0` num arquivo `.rs` real.
- Todos os 10 testes de `agent_loop::tests` continuam verdes.

**Impacto histórico:**
- LSP / DAP / Browser **NUNCA** funcionaram em produção desde a introdução do `build_registry` interno em `AgentLoop`.
- O snapshot test `default_registry_tool_id_snapshot_is_pinned` PASSAVA porque pinava o registry vazio (correto na sua intenção, mas não detectava que a versão real nunca era usada).

### F3 🟠 HIGH — Browser sidecar resolution só funciona dentro do checkout do theo

**Localização:** `crates/theo-tooling/src/registry/mod.rs::resolve_browser_sidecar_script`

```rust
let in_repo = project_dir.join("crates/theo-tooling/scripts/playwright_sidecar.js");
if in_repo.exists() { return in_repo; }
project_dir.join(".theo/playwright_sidecar.js")  // fallback nunca existe
```

**Problema:**
- Quando o usuário roda `theo --repo /tmp/meu-projeto`, project_dir não é o checkout do theo. O caminho `<projeto>/crates/theo-tooling/scripts/playwright_sidecar.js` não existe.
- Fallback `<projeto>/.theo/playwright_sidecar.js` só funciona se o usuário copiar manualmente.
- Resultado: Browser inutilizável fora do dev workflow do próprio theo.

**Recomendação:**
1. Embarcar o sidecar (`include_str!`) no binário e materializar em `~/.cache/theo/playwright_sidecar.js` no startup.
2. OU criar `theo browser install` que copia o sidecar + instala playwright npm + chromium.
3. OU resolver via `cargo` build script que coloca em `target/release/share/`.

### F4 🟡 MEDIUM — `theo checkpoints list` retorna mensagem confusa

**Repro:**
```bash
$ theo checkpoints list
Error: git command failed (exit 128): fatal: your current branch 'master' does not have any commits yet
```

**Esperado:**
```
No checkpoints recorded yet.
```

**Causa:** o shadow git repo em `~/.theo/checkpoints/<sha16>/` foi criado vazio mas nenhum commit/checkpoint foi registrado; o erro do `git log` (do shadow repo) é exposto cru.

**Fix sugerido:** detectar "branch has no commits" como caso degenerado e printar mensagem amigável.

### F5 🟡 MEDIUM — `cargo clippy --all-targets -- -D warnings` falha em bench

**Local:** `crates/theo-agent-runtime/benches/run_engine_bench.rs:127`
```rust
&format!("call-{i}"),
^^^^^^^^^^^^^^^^^^^^ help: change this to: format!("call-{i}")
```

**Causa:** o gate canônico `make check-sota-dod` roda clippy com `--lib --tests --bins` (sem `--benches`). Os arquivos sob `benches/` portanto não são gateados.

**Recomendação:** ou (a) corrigir o `&format!`, ou (b) adicionar explicitamente `--benches` ao SOTA gate para incluir bench files.

### F8 🟢 LOW — Sidecar Playwright requer node_modules co-localizado

**Workaround validado:** `cd crates/theo-tooling/scripts && npm install playwright`.

**Recomendação:** se F3 for resolvido (sidecar materializado em `~/.cache/theo`), instalar playwright no mesmo dir.
