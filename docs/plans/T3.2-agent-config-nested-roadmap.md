## Why this is its own document

`AgentConfig` (in `crates/theo-agent-runtime/src/config/mod.rs:198`)
has 45 flat fields covering 8 logical sub-domains. Phase T4.1 added
**view structs** (`config/views.rs`) as a temporary bridge:
`config.llm()`, `config.loop_cfg()`, etc., return short-lived view
objects holding references to the flat fields. T3.2 was supposed to
finish the work by promoting those views to **owned nested structs**
and deleting the views file.

Doing this in one PR would touch:

- 45 fields in `AgentConfig` definition
- 7 view-method call sites returning views (currently `config.llm()`)
- 50 call sites that consume views (`config.llm().model`,
  `config.loop_cfg().max_iterations`, etc.)
- ~30 test fixtures that build `AgentConfig` with field-by-field
  literals or `..Default::default()` spreads

Same risk profile as T3.1 — the right path is incremental.

## PR sequence

### PR1 — `LlmConfig` (the simplest cluster)

1. Define `pub struct LlmConfig { base_url, api_key, model, … }` in
   `crates/theo-agent-runtime/src/config/llm_config.rs` (NEW).
2. Add `pub llm: LlmConfig` field to `AgentConfig`.
3. Convert `LlmView<'a>` to a thin `pub fn llm(&self) -> &LlmConfig`
   getter. Keep call sites working unchanged.
4. Migrate `Default for AgentConfig` to nest the LLM defaults.
5. Update test fixtures: replace
   `AgentConfig { base_url: ..., api_key: ..., .. }` literals with
   `AgentConfig { llm: LlmConfig { base_url, api_key, .. }, .. }`
   wrappers.

### PR2-PR7 — One per remaining sub-domain

`LoopConfig`, `ContextConfig`, `MemoryConfig`, `EvolutionConfig`,
`RoutingConfig`, `PluginConfig` — same pattern as PR1.

### PR8 — Delete `config/views.rs`

After all 7 sub-configs are migrated, the views file holds only
deprecated wrappers. Delete it and inline any remaining call sites
to the owned getters.

## Why we did NOT land this in the Ralph loop

Same reasoning as T3.1's roadmap doc: each sub-config migration is a
bisect-friendly atomic change that wants to be its own PR with
co-ordinated test-fixture updates. The Ralph loop completed all
foundation work that depended on `AgentConfig` (capability gate
default-on, fence_untrusted wiring, etc.) — those landed using the
existing flat structure. T3.2 is now an independent cleanup that can
proceed in 8 PRs without blocking anything else.
