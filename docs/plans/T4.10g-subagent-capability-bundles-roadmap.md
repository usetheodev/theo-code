# T4.10g — `SubAgentManager` Optional → Typed Capability Bundles

> **Parent plan:** [agent-runtime-remediation-plan.md](./agent-runtime-remediation-plan.md) §T4.10g
> **Finding:** find_p3_006 — 12 `Option<…>` fields where invalid
> combinations cannot be ruled out at the type level (e.g.
> `mcp_discovery` without `mcp_registry`).
> **Status:** PENDING — documented as multi-PR roadmap to land
> alongside the broader god-object split (T3.1) since both refactor
> the same struct family.

---

## Why this is its own document (and a multi-PR effort)

`SubAgentManager` (in `crates/theo-agent-runtime/src/subagent/mod.rs:68`)
holds 12 `Option<Arc<…>>` fields plus the builder methods that set
them one by one. The current shape allows representable but invalid
states:

```rust
SubAgentManager {
    mcp_registry: None,
    mcp_discovery: Some(_),  // ← discovery without a registry to discover into
    ...
}
```

The plan's recommendation is to **bundle related Options into typed
capability structs** so invalid combinations become compile errors:

```rust
pub struct McpCapabilities {
    pub registry: Arc<McpRegistry>,
    pub discovery: Option<Arc<DiscoveryCache>>,
}
pub struct SubAgentManager {
    mcp: Option<McpCapabilities>, // either both/neither — discovery alone unrepresentable
    ...
}
```

The migration is the same shape as T3.1 god-object split: each
bundle changes the struct + its builder + ~50 test fixtures + the
spawn_with_spec consumers. Bisect-friendly only if done one bundle
per PR.

---

## Proposed bundles

| Bundle | Fields today | Invariant enforced |
|---|---|---|
| `McpCapabilities` | `mcp_registry`, `mcp_discovery` | discovery requires registry |
| `SubagentLifecycle` | `cancellation`, `checkpoint_manager`, `worktree_provider` | worktree requires checkpoint (current convention) |
| `SubagentObservability` | `metrics`, `hook_manager` | metrics + hooks fire together |
| `SubagentPersistence` | `run_store` | (1 field — keep flat) |
| `SubagentRouting` | `registry`, `pending_resume_context` | (related to dispatch) |

After bundling, the `SubAgentManager` struct shrinks from 12 fields
to ~6 (4 bundles + 2 standalone like `spawn_semaphore` and `depth`).

## PR sequence

### PR1 — `McpCapabilities` (the most natural bundle)

- `crates/theo-agent-runtime/src/subagent/capabilities/mcp.rs` (NEW) — define `McpCapabilities { registry, discovery }`.
- Replace `mcp_registry: Option<Arc<McpRegistry>>` and
  `mcp_discovery: Option<Arc<DiscoveryCache>>` with a single
  `mcp: Option<McpCapabilities>` field on `SubAgentManager`.
- Builder: collapse `with_mcp_registry(...)` and
  `with_mcp_discovery(...)` into `with_mcp(McpCapabilities)`. Provide
  back-compat shim `with_mcp_registry_only(reg)` constructing
  `McpCapabilities { registry: reg, discovery: None }`.
- Update `spawn_with_spec` and `register_mcp_tools` to consume
  `&self.mcp` instead of two separate Options.
- Update ~17 test fixtures.

### PR2-PR4 — Same pattern for the other bundles

`SubagentLifecycle`, `SubagentObservability`, and the routing pair.

### PR5 — Final trim

After all bundles land, audit remaining flat Options. `depth` and
`spawn_semaphore` are pure scalars — keep as flat fields. Remove
back-compat shims if no external consumer uses them.

## Why we did NOT land this in the Ralph loop

T4.10g shares the same multi-PR rationale as T3.1 and T3.2:

1. **Each capability bundle wants its own bisect-friendly PR** —
   landing them in one diff produces a 1000+ LOC change touching the
   manager + builders + 17+ test fixtures, which review tools cannot
   evaluate effectively.
2. **Builder API is part of the public surface** — back-compat shims
   need a deprecation cycle. The Ralph loop can't wait for that.
3. **Coupled with T3.1** — the SubAgentManager refactor logically
   pairs with the AgentRunEngine `SubagentContext` extraction
   (T3.1 PR1). Doing both together gives a single coherent
   refactor of the subagent surface.

Recommended sequence post-loop:
- Land T3.1 PR1 (SubagentContext) first.
- Land T4.10g PR1 (McpCapabilities) immediately after, taking
  advantage of the now-simpler manager surface.
- Continue T3.1 PR2-5 and T4.10g PR2-5 interleaved as the same
  team's natural workflow.
